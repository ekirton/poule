# Textbook RAG — Developer Guide

The textbook RAG system provides retrieval-augmented generation over the [Software Foundations](https://softwarefoundations.cis.upenn.edu) textbook series. It powers the `/textbook` slash command and the `education_context` MCP tool.

## Architecture Overview

```
software-foundations/          HTML source (7 volumes, ~150 pages)
        │
        ▼
HTMLChunker                    Section-based chunking (~1000 tokens/chunk)
        │
        ▼
EducationEncoder               all-MiniLM-L6-v2 (INT8 ONNX, 384-dim)
        │
        ▼
education.db                   SQLite: chunks + embeddings + FTS5 + metadata
        │
        ▼
EducationRAG                   In-memory cosine search via EmbeddingIndex
        │
        ▼
education_context MCP tool     Returns passages with citations + browser paths
```

**Key files:**

| File | Purpose |
|------|---------|
| `src/Poule/education/chunker.py` | HTML parsing + section-based chunking |
| `src/Poule/education/encoder.py` | all-MiniLM ONNX wrapper (384-dim, INT8) |
| `src/Poule/education/storage.py` | SQLite schema + read/write |
| `src/Poule/education/build.py` | Build pipeline: HTML → chunks → embeddings → DB |
| `src/Poule/education/__init__.py` | `EducationRAG` facade (query-time) |
| `src/Poule/education/handler.py` | MCP tool handler |
| `commands/textbook.md` | `/textbook` slash command |

## Updating When a New SF Version Is Released

Software Foundations releases are not tied to Coq/Rocq versions. When a new SF version comes out, follow these steps to update the bundled education database.

### 1. Download the new SF release

From the project root, run the download script:

```bash
./scripts/download-software-foundations.sh --force
```

This downloads all 7 volumes as tarballs from [softwarefoundations.cis.upenn.edu](https://softwarefoundations.cis.upenn.edu) and unpacks them into `software-foundations/`. The `--force` flag re-downloads volumes that already exist locally.

The resulting directory structure:

```
software-foundations/
├── lf/          # Logical Foundations
├── plf/         # Programming Language Foundations
├── vfa/         # Verified Functional Algorithms
├── qc/          # QuickChick
├── secf/        # Security Foundations
├── slf/         # Separation Logic Foundations
└── vc/          # Verifiable C
```

Each volume directory contains `.html` files generated by coqdoc plus a `common/` subdirectory with CSS/JS assets. The `software-foundations/` directory is `.gitignore`'d — it is downloaded at Docker build time and during local development.

### 2. Rebuild the education database

Inside the dev container:

```bash
poule-dev
```

Then run the build pipeline:

```bash
uv run python -m Poule.education.build \
    --sf-dir /poule/software-foundations \
    --output /data/education.db \
    --model /data/models/education/encoder.onnx \
    --tokenizer /data/models/education/tokenizer.json
```

This will:
1. Parse all HTML files across 7 volumes (skipping `index.html`, `toc.html`, `coqindex.html`, `deps.html`)
2. Chunk content at section boundaries (~1000 tokens per chunk)
3. Encode all chunks with the all-MiniLM-L6-v2 model
4. Write chunks, embeddings, and FTS5 index to `education.db`
5. Record metadata (schema version, model hash, build date, chunk count)

The pipeline prints statistics when complete:

```
Done: {'total_chunks': 847, 'total_tokens': 612340, 'volumes_indexed': 7}
```

### 3. Verify retrieval quality

Run the evaluation harness to confirm retrieval quality hasn't degraded:

```bash
uv run python -m Poule.education.evaluate \
    --db /data/education.db \
    --model /data/models/education/encoder.onnx \
    --tokenizer /data/models/education/tokenizer.json
```

Check that recall@3 at chapter level exceeds 80%.

### 4. Test the MCP tool

Restart the MCP server and verify the tool works:

```bash
poule-mcp restart
```

Then in Claude Code:

```
/textbook how does induction work in Coq?
```

Verify that results are returned with correct citations and browser paths.

### 5. Rebuild the Docker image

The production Dockerfile copies `software-foundations/` into the image and runs the build pipeline at build time. Rebuild the production image to bake in the new content:

```bash
docker build -t poule:latest .
```

The Dockerfile will:
1. COPY `software-foundations/` to `/poule/software-foundations/`
2. COPY model files to `/data/models/education/`
3. RUN the build pipeline to produce `/data/education.db`
4. The SF HTML stays in the image at `/poule/software-foundations/` and is copied to `$HOME/software-foundations/` at container startup for browser access

### 6. Commit and release

Commit the updated `software-foundations/` directory and push. The new container image will include the updated textbook content and pre-built education database.

## Model Files

The encoder model and tokenizer are not checked into the repository. They must be present at these paths in the Docker image:

| File | Path in container | Size |
|------|------------------|------|
| all-MiniLM-L6-v2 (INT8 ONNX) | `/data/models/education/encoder.onnx` | ~23 MB |
| WordPiece tokenizer | `/data/models/education/tokenizer.json` | ~700 KB |

To obtain these files, run the download script from the project root (on the host, not inside the container):

```bash
./scripts/download-education-model.sh
```

This creates a temporary Python venv, installs `optimum` and `onnxruntime`, exports the model to ONNX, quantizes to INT8, and copies the results to `models/education/`. The temp venv is cleaned up automatically.

The `models/education/` directory is `.gitignore`'d — model files are not checked into the repository.

## Database Schema

The education database (`education.db`) contains:

| Table | Purpose |
|-------|---------|
| `chunks` | Chunk text + metadata (volume, chapter, section, anchor ID) |
| `education_embeddings` | 384-dim float32 vectors (1536 bytes each) |
| `education_meta` | Key-value metadata (schema_version, model_hash, build_date, chunk_count) |
| `chunks_fts` | FTS5 full-text index for keyword fallback search |

## Chunking Details

The chunker splits coqdoc-generated HTML at `<h1-h4 class="section">` boundaries. Size control:

- **Target:** ~1000 tokens per chunk
- **Split:** Sections over ~1000 tokens split at `<div class="doc">` (paragraph) boundaries
- **Merge:** Sections under 100 tokens merge with the following section
- **Exception:** Exercise sections (`<h4>` containing "Exercise") are never merged

Each chunk preserves:
- Plain text (prose + inline code)
- Coq code blocks as separate strings (for display)
- Source attribution: volume, chapter, section title, section breadcrumb path, anchor ID
- Chapter filename (for browser link construction)

## Troubleshooting

**education_context returns EDUCATION_UNAVAILABLE:**
The education database or model files are not present. Check that `/data/education.db`, `/data/models/education/encoder.onnx`, and `/data/models/education/tokenizer.json` all exist.

**Low retrieval quality after SF update:**
The HTML structure may have changed between SF versions. Check the chunker output for empty or malformed chunks:

```python
from Poule.education.chunker import HTMLChunker
chunker = HTMLChunker()
chunks = chunker.chunk_corpus("/poule/software-foundations")
for c in chunks:
    if c.token_count < 10:
        print(f"Tiny chunk: {c.metadata.volume}/{c.metadata.chapter_file} — {c.metadata.section_title}")
```

**Model hash mismatch warning:**
The `education_meta.model_hash` records which encoder produced the embeddings. If you replace the ONNX model, rebuild the entire database — you cannot mix embeddings from different models.
