- Read the upstream PRD before writing or editing.
- Describe the feature from the user's perspective — what it does, why it exists, and the design tradeoffs.
- Do not describe pipelines, data formats, or implementation mechanics — those belong in architecture.
- State what the feature provides and what it explicitly does not provide.
- Include an Acceptance Criteria section with GIVEN/WHEN/THEN entries tracing to PRD requirement IDs.
- Priority levels: P0 = must-have, P1 = should-have, P2 = nice-to-have, P3 = future consideration.
